The '''Ecstasy''' programming language (aka '''xtclang''') is designed to support the efficient composition of secure applications, long-term maintainable infrastructure, and efficient, low-cost deployments in the serverless cloud. The language was originally created by xqiz.it within the open source Ecstasy Project community (hosted on Github and Discord), and has been contributed to by dozens of developers from across the industry. It is licensed under the business-friendly Apache 2 open source license, and its standardization and evolution is overseen by the Ecstasy Foundation.

Ecstasy source code is written in one or more <code>.x</code> source files, [[Compiler|compiled]] to an intermediate <code>.xtc</code> [[bytecode]] executable file using the <code>xcc</code> command, and executed by the XTC Virtual Machine (XVM) using the <code>xec</code> command. The bytecode is a mid-level IR designed to be efficiently lowered to native machine code or other target IRs by any combination of ahead-of-time, just-in-time (JIT), and adaptive/optimizing JIT compilation. The compiler, interpreter, and JIT runtime are included in the Ecstasy Development Kit (XDK).

The goals of Ecstasy are:
* Source code that is easy to read and understand; a language that is easy to learn for any developers already familiar with widely-used languages like: Java, C#, Python, Swift, Kotlin, and C++; and composition capabilities that make it simultaneously efficient and powerful for application and service developers.
* A hierarchical container model designed for the secure serverless cloud, using lightweight in-process containers and dependency injection (IOC/DI), and container-level resource management (CPU, memory, storage, network, etc.)
* A fully reified generic type system with support for unions, intersection, difference types; first-class tuples, functions, and modules; bi-directional type inference; and type-safe functional mixins.
* Built-in memory-safe concurrency, using a concurrent service model coupled with a managed fiber-based runtime, with first-class support for deep immutability.
* Built-in module versioning, including: full version-aware dependency resolution, optional dependency support, module repository support, and multi-version compiled modules.
* All of the modern capabilities that developers have grown to depend on, such as: [[garbage collection|Automatic memory management]], memory safety, an integrated module system, first-class lambdas and functions, partial binding and currying, cross-platform portability, and an open source development model.
